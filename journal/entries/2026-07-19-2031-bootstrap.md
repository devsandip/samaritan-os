# 2026-07-19 20:31 — v0 built in one sitting, from design docs to a working review gate

## What happened

Samaritan went from a folder of design documents to running software today. This
morning the repo held seven markdown files and three HTML mockups. Tonight it
holds a working Action Center, 119 tests, and a public GitHub repo.

The design suite was already done and internally consistent. `TECH-SPEC.md` has
a build order in section 12, fifteen numbered steps for v0. I worked through it
in order. The contracts first, because everything depends on them. Then the
store, the policy engine, the registry, routing, execution, ingest. Then the
anchor.

The anchor is the thing that matters. `wrap` and `meeting` used to extract
structured items from a conversation and write them straight to Notion and
TickTick. Now they extract exactly as before and pipe the result to
`samaritan emit`, and nothing is filed until I approve it. That is the whole v0
thesis: a human review gate on the highest-inference capabilities, proven on
two real ones before generalising.

I verified it end to end three times, at increasing levels of realism. First in
unit tests. Then against a live local server writing to a scratch vault. Then
against my actual Notion workspace, with real rows created and archived
afterwards. Each level found bugs the level below could not have.

## What broke and what it taught me

Six real bugs today. Only two came from tests.

`better-sqlite3` has no prebuilt binary for Node 26 and pnpm would not compile
it. Switched to Node's built-in `node:sqlite`, which does everything section 3
actually asked for. That one surfaced in the first five minutes.

The expr-eval sandbox was not a sandbox. Clearing `parser.functions` left
`sqrt`, `abs` and friends behind, because expr-eval registers them as unary
*operators* in a different table. A test caught it. The fix was to rebuild all
three built-in tables from an allowlist rather than remove known-bad names one
at a time.

`pm-os.item.file` forwarded the item payload unchanged to whichever adapter it
dispatched to. That only works when the shapes happen to agree, and they never
did. Notion wants `rationale`, Obsidian wants `path` and `content`. The live run
caught it, and it only surfaced because strict validation rejected the
undeclared keys instead of dropping them quietly.

The vault root was wrong. Config pointed at `~/Documents/Obsidian`, but the
vault is `~/Documents/Obsidian/Samaritan`. Daily notes would have landed in a
sibling folder Obsidian never indexes. Two call sites disagreed about which one
carried the `Samaritan` segment, so both halves were self-consistently wrong and
no test would ever have caught it. I found it by checking the running skills
against the config.

The Notion adapters had the wrong property names, twice. First I wrote them from
the SKILL.md summaries and got `Name` where the schema says `Decision`. Then I
corrected them against `AGENT_OS.md` in the vault and still had `Decided on`
instead of `Decided On`, plus a write to `Captured On`, which is a
`created_time` property Notion computes and refuses to accept. Only the live API
call found those.

The worst one: the database ids in `AGENT_OS.md` are Notion **data source** ids,
not **database** ids. The doc lists both, a `data_source_id` column and a URL
carrying a different value. The Notion MCP tool takes the first. The REST API
takes the second. Every write would have 404'd. I now get ids by asking the API
rather than reading them out of a doc.

The pattern is consistent. Tests catch logic errors. Only contact with the real
system catches integration errors. I should stop treating a green suite as
evidence that something works.

## Where we are now

v0 is complete and verified. Ten commits, public at
github.com/devsandip/samaritan-os.

The full loop runs: a capability emits, the Policy Engine escalates, the item
waits in the Inbox, I approve or edit-then-approve or reject or defer, and only
then does anything reach Notion, TickTick or the vault. Every transition writes
an audit row naming its actor. The Inbox UI is a React SPA with renderers
dispatched off `render.layout`, so nothing in the UI knows the name of any
capability.

Three properties are enforced structurally rather than by convention. A status
change with no audit row is not representable, because `transition()` writes
both in one transaction and SQLite triggers abort any update or delete on the
audit table. Money never moves automatically, checked in three independent
places that must all agree. Nothing has no fallback, because a missing or
incapable adapter degrades to a guided path that renders the action as
copy-ready text.

Notion is live. Token authenticates, all four databases connected, ids correct,
property names verified against real writes including the Project relation
lookup. Telegram is written but parked and disabled.

## What I believe

Gating the highest-inference capabilities first was the right call. `wrap` and
`meeting` are where a wrong row does the most damage, and they were the two
places I had the least visibility. Everything else generalises down from there.

The manifest-driven design is holding. Adding a capability really is dropping a
folder. I did not touch Action Center code to add the second one.

Strict validation earns its cost. Rejecting undeclared keys rather than
stripping them turned a silent data-shape mismatch into a loud one, twice.

Publishing forced a discipline I would not have applied otherwise. Scrubbing
only HEAD would have looked thorough and been cosmetic, since the vendored
plugin commit still had the real ids in its tree.

## Open questions

A deferred item is currently a dead end. Nothing sweeps it back to pending and
`respond` refuses every response on it, so "act now" 409s. This is the most
visible gap.

An orphaned item cannot be actioned at all. If a capability is unloaded after
ingest, every response returns `response_unknown` and there is no universal
response id the server accepts.

Recall is unbuilt. `src/recall/` is empty, so Ask-Samaritan renders as a
placeholder in the UI.

Whether one action-item type per capability with a `kind` discriminator was
right, or whether four separate types would have been cleaner. It works, and the
dispatching adapter is a routing table rather than a second implementation, but
I am not sure yet.
